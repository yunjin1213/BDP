# Spark와 Hive를 활용한 서울 관광지 방문 패턴 분석 및 상황별 관광 코스 추천

**부제: 생활인구·지하철·기상·대기질 데이터 융합 분석**

## 1. 프로젝트 개요

본 프로젝트는 서울을 방문한 친구, 데이트 상대, 또는 외부 방문객과의 코스를 계획할 때 날씨, 미세먼지, 시간대, 혼잡도 등을 함께 고려해야 하는 번거로움을 데이터 분석으로 줄이는 것을 목표로 한다.

서울 주요 관광지 주변 지역에 대한 생활인구, 지하철 승하차, 기상, 대기질 데이터를 수집하고, HDFS에 적재한 뒤 Spark와 Hive를 활용해 관광지별 방문 패턴과 환경 민감도를 분석한다. 이후 분석 결과를 기반으로 특정 상황에 적합한 관광지 및 간단한 관광 코스를 추천한다.

본 프로젝트의 핵심은 추천 서비스 구현이 아니라, Spark/Hive를 활용해 관광지별 방문 패턴과 환경 민감도를 분석하는 것이다. 코스 추천은 분석 결과를 활용한 응용 결과물로 제시한다.

## 2. 문제 정의

서울에는 경복궁, 북촌, 홍대, 성수, 잠실, 명동 등 잘 알려진 관광지와 데이트 코스가 많다. 하지만 실제로 코스를 정할 때는 단순히 유명한 장소인지보다 다음과 같은 요소가 중요하다.

- 비가 오는지, 너무 덥거나 추운지
- 미세먼지가 심한지
- 특정 시간대에 사람이 너무 붐비는지
- 실내/실외 장소를 어떻게 조합할지
- 이동 동선이 너무 복잡하지 않은지

따라서 본 프로젝트는 단순 인기 관광지 추천이 아니라, 실제 방문 패턴 데이터를 기반으로 관광지별 환경 민감도와 시간대별 혼잡도를 분석하고, 이를 활용해 날씨·미세먼지·시간대·혼잡도를 고려한 상황별 서울 관광 코스 추천 기준을 제안한다.

## 3. 핵심 분석 질문

1. 날씨와 미세먼지 조건에 따라 서울 관광지 주변 지역의 방문 패턴은 어떻게 달라지는가?
2. 시간대별로 혼잡도가 높은 관광지와 비교적 여유로운 관광지는 어디인가?
3. 환경 변화에 민감한 관광지와 안정적인 관광지는 어디인가?
4. 분석 결과를 활용하면 특정 상황에 어떤 관광 코스를 추천할 수 있는가?

## 4. 사용 데이터 및 API

| 데이터 | 출처 | 활용 목적 |
| --- | --- | --- |
| 관광지 정보 | 한국관광공사 국문 관광정보 서비스 | 관광지명, 주소, 좌표, 관광지 유형 수집 |
| 주소/좌표 변환 | 스마트서울맵 OpenAPI | 관광지 주소 또는 좌표를 행정동으로 변환 |
| 서울 생활인구 | 서울 열린데이터광장 | 관광지 주변 행정동의 시간대별 체류 인구 분석 |
| 지하철 승하차 | 서울 열린데이터광장 | 관광지 주변역의 시간대별 유입량 분석 |
| 지하철역 위치 | 서울 열린데이터광장 | 관광지 좌표와 가까운 주변역 연결 |
| 기상 관측 | 기상청 ASOS 관측자료 | 기온, 강수량 등 날씨 조건 분류 |
| 대기질 | 서울 대기환경정보 API | PM10, PM2.5 기반 미세먼지 조건 분류 |

초기 분석 대상은 경복궁/북촌, 명동, 홍대, 성수, 잠실 등 5개 내외의 주요 관광 권역으로 제한한다. 각 관광지는 관광지 정보 API에서 제공하는 주소·좌표 데이터를 활용해 행정동에 매핑하고, 지하철역 위치 데이터와 결합하여 주변역을 연결한다.

누적 100MB 이상의 데이터를 확보하기 위해 최소 1~3개월 이상의 기간을 대상으로 데이터를 수집한다. 수집된 원천 데이터는 JSON 또는 CSV 형태로 보존하며, 이후 HDFS에 적재한다.

## 5. 시스템 아키텍처

```text
OpenAPI 데이터 수집
  - 관광지 정보 API
  - 주소/좌표 변환 API
  - 서울 생활인구 API
  - 지하철 승하차 API
  - 기상청 관측 API
  - 서울 대기질 API
        ↓
Raw Data 저장
  - JSON 또는 CSV 형태로 로컬 저장
        ↓

  - 원천 데이터를 HDFS에 업로드
        ↓
Spark 전처리
  - 날짜/시간 컬럼 정리
  - 결측치 및 이상치 처리
  - 관광지-행정동-주변역 매핑
  - 날씨/미세먼지/시간대 조건 생성
        ↓
Hive 분석 테이블 생성
  - 관광지별 방문량 테이블
  - 시간대별 혼잡도 테이블
  - 환경 민감도 테이블
        ↓
Spark SQL / HiveQL 분석
  - GROUP BY, JOIN, 변화율, 통계 분석
        ↓
결과 시각화 및 추천
  - 그래프 생성
  - 상황별 관광지/코스 추천
```

## 6. 데이터 처리 및 분석 방법

Spark DataFrame을 사용해 원천 데이터를 정제하고 분석 가능한 형태로 변환한다.

주요 전처리 과정은 다음과 같다.

- 날짜, 시간대 컬럼 형식 통일
- 행정동, 자치구, 지하철역 명칭 정리
- 결측치 및 이상치 처리
- 관광지-행정동-주변역 매핑 테이블과 조인
- 시간대 구간 생성: 오전, 오후, 저녁, 야간
- 날씨 조건 생성: 비, 맑음, 더움, 추움
- 미세먼지 조건 생성: 좋음, 보통, 나쁨
- 관광지별 방문량 및 혼잡도 집계

분석은 Spark SQL과 HiveQL을 활용한다. 단순 조회가 아니라 `GROUP BY`, `JOIN`, 평균, 변화율, 표준편차 등을 활용해 관광지별 방문 패턴과 환경 민감도를 계산한다.

### 시간대별 혼잡도

```text
Congestion Index =
Tourist Spot Visit / Average Visit of All Tourist Spots in Same Time Slot
```

같은 시간대의 다른 관광지 평균 방문량과 비교하여 특정 관광지가 상대적으로 얼마나 붐비는지 계산한다.

### 환경 조건별 방문 변화율

```text
Visit Change Rate =
(Condition Avg Visit - Baseline Avg Visit) / Baseline Avg Visit
```

Condition Avg Visit은 특정 환경 조건에서의 평균 방문량을 의미한다. 예를 들어 비 오는 날, 고온일, 저온일, 미세먼지 나쁨 일자의 평균 방문량이 이에 해당한다. Baseline Avg Visit은 해당 조건을 제외한 일반 상황의 평균 방문량을 의미한다.

이 지표를 통해 각 관광지가 특정 환경 조건에서 평상시 대비 얼마나 방문량이 증가하거나 감소하는지 비교한다.

### 방문 안정성

```text
Stability =
1 / Coefficient of Variation
```

시간대별 방문량의 변동계수를 활용하여 특정 관광지가 시간대나 환경 조건 변화에도 안정적으로 방문되는지 계산한다.

## 7. 추천 방식

추천은 임의의 규칙이나 외부 추천 API에 의존하지 않고, Spark/Hive 분석을 통해 계산한 관광지별 시간대 혼잡도, 환경 조건별 방문 변화율, 방문 안정성 지표를 활용한다.

즉, 추천 결과는 분석 단계에서 생성된 지표 테이블을 기반으로 산출되며, LLM이나 외부 서비스의 일반 지식에 의존하지 않는다.

예를 들어:

- 비에 민감한 관광지는 비 오는 날 추천에서 제외
- 미세먼지에 영향을 적게 받는 실내/상권형 장소는 미세먼지 나쁨 조건에서 우선 추천
- 특정 시간대에 지나치게 혼잡한 장소는 감점
- 같은 권역 내 가까운 관광지를 묶어 2~3개 장소 코스로 구성

## 11. 확장 가능성

본 프로젝트는 기본적으로 서울 관광지 주변 방문 패턴 분석에 초점을 맞추지만, 다음과 같이 확장할 수 있다.

- 실시간 API를 활용한 현재 상황 기반 코스 추천
- LLM 기반 자연어 관광 코스 추천
- 지도 API를 활용한 이동 거리 및 이동 시간 반영
- 사용자 선호 기반 개인화 추천
- 음식점, 카페, 전시, 공연 데이터와 결합
- 카드 매출 또는 상권 데이터와 결합한 소비 패턴 분석
- 웹 대시보드 또는 서비스 구현
- 머신러닝 기반 방문량 예측 모델 추가

LLM은 추천 후보를 직접 결정하는 주체가 아니라, 분석 결과와 실시간 데이터를 바탕으로 사용자의 자연어 요청을 해석하고 추천 이유를 설명하는 인터페이스 역할을 한다.

## 12. 한계

- 관광지 방문객 수를 직접 측정하는 것이 아니라, 생활인구와 지하철 승하차 데이터를 대리 지표로 사용한다.
- 특정 지역의 생활인구에는 관광객뿐 아니라 거주자, 직장인, 통행 인구가 함께 포함될 수 있다.
- 관광지와 행정동/지하철역 매핑 과정에서 일부 오차가 발생할 수 있다.
- 날씨와 방문량의 관계는 상관관계 분석이며, 인과관계를 직접 증명하는 것은 아니다.
- 추천 코스는 실제 개인 취향, 이동 수단, 예산, 동행자 특성을 완전히 반영하지는 않는다.
- 공공 API의 제공 기간, 호출 제한, 결측 데이터 여부에 따라 분석 가능한 기간과 세부 지역 범위가 달라질 수 있다.
